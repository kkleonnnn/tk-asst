# 角色分工（团队定稿 v3）

> **适用**：TikTok Shop 马来西亚跨境店项目，配套《从 0 到稳定出单执行计划》使用。
> **时间**：2026-06-21（v3：两遍消化制——第一遍人人可做、最终整合归 kk）。属"活文档"，分工有变就改这里。
> **一句话**：**校长管决策/审核/关系/情报，kk 把重复执行自动化掉 + 做最终整合，AI 是产出与消化引擎，体力活外包。**

---

## 一、三个角色

- **校长 = 运营负责人（决策 · 审核 · 关系 · 情报）**
  他在跨境电商课程平台工作，**情报近水楼台**——平台政策、选品风向、打法、培训资料由他持续喂进来。职责聚焦：
  - **情报/喂料 + 第一遍消化**（主力）：把看到的政策/选品/打法**自己先用 AI 消化成 `.md`** 丢进 `inbox/`（不限模型）。
  - **运营决策**：选品方向、定价策略、活动策略、达人策略、GMV 目标。
  - **审核拍板**：listing/话术/活动方案的终审。
  - **关键沟通**：头部达人关系、客服升级与纠纷。
  - **他不亲手干重复执行**（批量上架/批量建联/一线客服/拉数据/自拍自播）——那些被自动化/AI/外包吃掉。
- **kk = 技术负责人（整合 · 自动化 · 基建）**
  - **最终整合**：把 inbox 里的第一遍 `.md` **校对/去重/核实/规范化后并入 `reference/`**（建议 Opus 4.8+ 把关）、**审 PR 合并 main**。
  - **自动化校长的重复活**：RPA 批量上架/批量建联、客服自动回复配置、数据看板、发货打单工具、选品打分器、利润计算器。
  - **注册经办 + 合规基建**：跑注册/后台配置/合规把关（**主体与账号归属待两人商议，见第三节**）、AI 编排。
- **AI（skill）= 产出工厂 + 消化引擎**
  listing/标题/详情/达人话术/FAQ/直播脚本/数据诊断**初稿**；知识消化。kk 调用与工具化，校长终审消费。**不碰真实账号/钱。**
- **外包**：发货→货代(3PL)；直播→达人/机构代播；短视频→达人 UGC。

> 大方向、预算额度 = **两人合议**。

---

## 二、给校长减负的四个杠杆（本版核心）

1. **kk 自动化吃掉重复点击**：批量上架、批量达人邀约、客服自动回复、数据看板——校长不用手动一个个干。
2. **AI 产初稿，校长只终审**：listing/话术/FAQ/脚本/分析全由 AI 出初稿。
3. **体力活外包**：发货交货代、直播找达人/机构代播、视频用达人 UGC。
4. **分阶段**：早期只做 MVP 必要执行；**短视频/直播/广告往后放**（计划 §十 本就说广告别早投）。

> ⚠️ 代价：**前期 kk 要先把自动化/工具搭起来（前置投入）**；搭好后校长日常很轻、可持续。

---

## 三、注册这块：经办归 kk，归属待两人商议

> **拆成两件事，别混**：① 这块活谁干（**经办**）② 店铺/主体/钱归谁（**归属**）。

- **经办（耗时的活）= kk**：准备材料、跑注册流程、后台配置、合规把关、对接平台。这块耗时，因工作原因 kk 来干合适。
- **归属（主体/账号/收款/法人/分账）= ⏳ 未定，两人优先商议**：用谁的营业执照/法人、收款挂谁名下、回款怎么分账。**注册前必须先定**（得先有"用谁的执照和法人"才能开店），见第六节第 1 项。

| 事项 | 经办（谁干） | 归属（⏳ 待商议） |
|---|---|---|
| 备齐执照/资质核对/法人证件/邮箱手机号/收款账户 | **kk** | 用谁的主体/法人/账户 |
| 注册流程 / 缴保证金 / 绑卡操作 | **kk 经办** | 钱从谁账户出、回款挂谁 |
| 法人实名/扫脸等绑定本人的步骤 | 最终确定的**法人本人**做 | 取决于归属 |
| 英文店铺名 | kk 出备选（管合规） | 校长从定位拍板 |
| 仓库/取件/退货地址 | 运营/物流（货代） | — |

## 四、按阶段分工（校长=拍板，执行=自动化/AI/外包）

| 计划阶段 / 事项 | 校长（决策·审核·关系·情报） | 执行（kk 自动化 / AI 产出 / 外包） |
|---|---|---|
| **情报 / 喂料 + 第一遍消化** | **主力**：自己先消化成 `.md` 丢 `inbox/`（不限模型） | kk 整合并入 `reference/` |
| **0 定位 + 选品** | 看 AI 排好的候选拍板 | AI 选品打分 + kk 选品器 |
| **0 材料 + 主体** | 归属待与 kk 商议 | **kk 经办备料（第三节）** |
| **1 注册/保证金/收款** | 归属待商议 | **kk 经办操作**；AI 备 checklist |
| **1 客服/装修** | 定客服规则、装修拍板 | AI 起草 FAQ/话术 → kk 配自动回复+模板 |
| **2 定价** | 定红线/策略 | kk 利润计算器自动算 |
| **2 上架** | 抽检终审 | **kk RPA 批量上架 + AI 生成 listing** |
| **3 达人** | 定策略 + 维护头部关系 | **kk RPA 批量邀约 + AI 话术**；寄样可走货代 |
| **3 客服** | 只处理升级/纠纷 | **自动回复机器人(kk 配) + AI 草稿** |
| **3 活动** | 拍板报哪些 | AI 方案 + 利润校验，kk 半自动报名 |
| **发货** | （可选）盯异常 | **货代外包** 或 kk 打单自动化 |
| **4–6 复盘/优化** | 看结论做决策 | **kk 数据看板 + AI 诊断报告** |
| **短视频/直播/广告** | 早期不做；稳定后定方向 | AI 脚本 + **达人 UGC / 代播**；广告延后 |
| **最终整合 / PR 合并** | — | **kk（把 inbox `.md` 并入 `reference/`，建议 Opus 4.8+；审 PR 合并 main）** |

> 校长净职责 ≈ **看排好的候选/报表做决策 + 终审 AI 初稿 + 维护关系 + 喂情报**；重复执行不落他手上。

## 五、skill 维护分工（两遍消化制）

- **第一遍消化 = 任何人 / 任何模型**：自备资料本地消化成 `.md` → `inbox/` → push/PR（**不限模型**，减轻 kk 负担）。
- **最终整合（inbox `.md` → `reference/`）= kk**：校对/去重/核实/规范化（建议 Opus 4.8+ 把关）。
- **PR 审核 + 合并 main = kk**（正式生效关口）；其他人走 PR、不直接推 main。
- **重生成 `通用知识包.md` = kk**（`scripts/build_pack.sh`）。

---

## 六、待你俩拍板的开放项

1. **⭐ 主体/账号/收款归属 + 分账（优先级最高，注册前必须定）**：用谁的营业执照/法人、收款挂谁名下、回款怎么分账、成本怎么记。**kk 只是经办（干活），归属未定。**
2. **发货**：自营打包 还是 **货代外包**？（建议外包，给校长减负 + 守 72h 入仓红线）
3. **直播**：自播 还是 **找达人/机构代播**？短视频是否走达人 UGC？
4. **广告预算**：额度上限、谁审批（默认 kk 批、校长提需求）。
5. **校长权限**：给 write 但**走 PR**（kk 审再合并），不直接推 main。
6. **kk 前置工作量**：自动化工具先搭哪几个（建议优先级：利润计算器 → 批量上架 → 达人批量建联 → 客服自动回复 → 数据看板）。

---

## 七、协作闭环

```
两人合议定大方向
  → 校长喂情报 + 第一遍消化成 .md 丢 inbox（不限模型）+ 做运营决策/终审/维护关系
  → kk 把 inbox 的 .md 整合并入 reference（最终质量关）+ 用 AI 把执行工具化
  → 重复执行由 自动化/AI/外包 承担，校长只拍板
  → kk 审 PR 合并 main → 知识库正式迭代 → 重生成通用包
```
